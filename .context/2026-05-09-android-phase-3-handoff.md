# Android Phase 3 — Closeout & Handoff for Phase 4/5 Design

**Date:** 2026-05-09
**Branch:** `docs/android-phase-3-plan` (31 commits ahead of `main`)
**Verified on:** Tab Active 5 (T865), Android 14, USB-OTG audio capture
**Run report:** `.context/2026-05-09-android-phase-3-results.md`

This document hands off everything a downstream design author needs to plan
Phase 4 (GPS), Phase 5 (TX/PTT), and the residual operator-UX polish.

## 1. What Phase 3 actually shipped

**Production Android entry that runs the full graywolf runtime in-process
to a Kotlin foreground Service.**

End-to-end RX path on T865:
1. Kotlin `AudioPump` (`MediaRecorder.AudioSource.MIC`) →
2. JNI `ModemBridge.modemPushSamples` →
3. Rust `graywolf-modem` cdylib (in-process to the Service) demods AFSK →
4. Frames published over Unix domain socket →
5. Go `cmd/graywolf` (forked from `Service` via `GoLauncher`) runs `app.New(cfg).Run(ctx)` →
6. modembridge in connect-only mode dials the existing UDS →
7. dispatcher → APRS decoder → station cache → history DB →
8. Web API serves SPA at `127.0.0.1:8080` with bearer-token middleware →
9. Kotlin `MainActivity` WebView loads it after Service signals `goListenerReady` →
10. SPA reads bearer token from JS bridge, injects into every fetch + WebSocket.

Live-verified APRS frames decoded through the full stack (NW5W-5, REDSPR,
KB7WHO-11, BLOW, MHILL, ELY, KF6RAL-* and dozens more).

### Architectural shifts vs phase 2

- **modembridge gained connect-only mode** (`pkg/modembridge/Config.ExistingSocket`).
  When set, supervisor skips `exec.CommandContext` + readiness handshake and dials
  the UDS directly. Phase 2 forked the modem child unconditionally; that doesn't
  work on Android where the modem cdylib is in-process to the Kotlin Service.
- **Bearer-token loopback auth** (`pkg/webauth/BearerAuthMiddleware`). Per-launch
  32-byte hex token generated by `GraywolfApp.onCreate` (per-process lifetime;
  stable across supervisor-driven Go-child restarts). Middleware gates `/api/`,
  `/ws/`, `/tiles/` only — static SPA paths (`/`, `/assets/*`) stay public so the
  WebView's first navigation can boot the SPA shell that then installs the bearer-
  injecting fetch wrapper.
- **Platform-aware wiring**. New `app.Config.Platform = "android"` gates:
  - `pkg/updatescheck.Checker` skipped (Play Store handles updates)
  - `ResolveModemPath` skipped when `ModemSocketPath != ""`
  - `PositionLogConfig` defaults to `Enabled=true` on first cold boot (the
    desktop SD-card protect rationale is moot on Android internal storage)
- **JS-bridge SPA integration**. `web/src/lib/androidBridge.js` reads the
  per-launch token from `globalThis.GraywolfWebInterface.getBearerToken()` once,
  caches. `secureFetch.js` patches `globalThis.fetch` and `globalThis.WebSocket`
  via `class extends WebSocket`. `bootstrap.js` is the first import in
  `main.js` so the patches land before any other module evaluates. `api.js`
  skips the `#/login` redirect on 401 when the bridge is present.
- **Modem cdylib publishes per-device level + StatusUpdate + tags frames
  with channel id.** New `graywolf-modem/src/android/config_state.rs`
  captures `ConfigureChannel.{channel, input_device_id}` from inbound IPC,
  plus an `AtomicU64 RX_FRAMES` cumulative counter. `audio.rs` computes
  peak/rms over a 200ms sliding window from the post-gain JNI samples,
  emits `DeviceLevelUpdate` at ~5Hz keyed by `input_device_id`. `mod.rs`
  tags every emitted `ReceivedFrame.channel` with the captured channel id
  (instead of the demodulator's internal `frame.chan` which is always 0
  for AFSK), increments `RX_FRAMES` per emission, and emits `StatusUpdate`
  at 1 Hz so the Go modembridge `status_cache` populates per-channel
  `ChannelStats.RxFrames` that the SPA Dashboard polls.

### Operator UX

- **Notification with Stop action** (`com.nw5w.graywolf.STOP` broadcast →
  `stopSelf()`). RECEIVER_NOT_EXPORTED so other apps can't fire it.
- **Battery-optimization whitelist intent** fired one-shot on first
  `MainActivity.onResume` (SharedPreferences flag prevents nagging).
- **WebView feels native**: pinch-zoom disabled, overscroll glow off,
  scrollbars off, long-press context menu swallowed.
- **SPA UI hidden where Android can't deliver**:
  - `web/src/lib/platform.js` exports `isAndroid()` / `isDesktop()`
  - AudioDevices "Detect Devices" + "Scan Input Levels" buttons hidden +
    explanatory note added
  - "Detected Hardware" grid hidden
  - Sidebar drops PTT (phase 5) and Simulation (desktop dev tool)

### Build pipeline

- `cmd/graywolf-pocb` retired. `goCrossCompile_*` Gradle tasks point at
  `cmd/graywolf`. Inputs include `web/dist` so the Go re-builds when the SPA
  bundle changes (without this, stale embeds shipped silently).
- Gradle wrapper jar repaired (phase 2 shipped a 42KB stub that crashed every
  invocation). Now 48KB, distribution pinned at 8.7 to match AGP 8.5.
- Cross-compile both ABIs clean: arm64 CGO_ENABLED=0 (~34M),
  amd64 CGO_ENABLED=1 via NDK (~34M).
- APK size: **48.9 MB** (vs phase-2 baseline 35.5 MB; +13.4 MB from
  production SPA embed).

## 3. Known gaps requiring downstream work

These are operator-visible deficiencies that phase 3's scope doesn't cover.
Phase 4/5 author should pull these into their plans.

### 3.1 No TX path (planned for phase 5)

`AudioPump` only does `AudioRecord` (input). No `AudioTrack` for output.
POC-C had a one-shot `AudioTxTest.fireOnce` that played a canned PCM buffer
through `AudioTrack`; that class still exists in tree but its JS-bridge
trigger was deliberately stripped (Task 9 of the phase-3 plan).

**Phase 5 needs:**
- Continuous `AudioTrack` writer fed by Rust modem TX (mirror of `AudioPump`'s
  RX shape: Rust pushes PCM samples via JNI, Kotlin streams to AudioTrack)
- PTT trigger path through proto: SPA → Go → modem cdylib → JNI → Kotlin
  USB device handler (CM108 HID, CDC-ACM RTS, AIOC firmware DTR)
- USB device permission flow (`android.hardware.usb.host` already declared,
  but `<intent-filter>` + `device-filter.xml` for auto-attach intent on
  Digirig/CM108 vendor:product IDs needed)
- Manifest re-add: `android.permission.FOREGROUND_SERVICE_CONNECTED_DEVICE`
  + a USB access perm (any of: USB_DEVICE permission, BLUETOOTH_*,
  CHANGE_NETWORK_STATE, CHANGE_WIFI_STATE) so `startForeground` accepts
  the `connectedDevice` FGS type
- `Service.startForeground` bitmap flips back to
  `MICROPHONE | CONNECTED_DEVICE`
- `WebAppInterface` regrows `requestUsbPermission(name, callbackId)` plus
  callback delivery via `webView.evaluateJavascript("window.__usbResult(...)")`

### 3.2 No GPS (planned for phase 4)

`pkg/gps/android.go` doesn't exist. Phase 2 shipped the platformsvc
SubscribeGpsFix surface and the Go-side already calls `cli.SubscribeGpsFix`
when configured. The Kotlin side has no `LocationManager` listener yet.

**Phase 4 needs:**
- `pkg/gps/android.go` (build-tagged) — consumes platformsvc fixes via the
  client, feeds the existing GPS reader interface
- Kotlin: new `GpsService` (or extension of `GraywolfService`) that owns a
  `LocationManager` `LocationListener`, translates `Location` callbacks into
  platformsvc `GpsFix` proto messages, broadcasts to subscribed clients
- Manifest: `<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION"/>`
  plus the runtime permission flow in `MainActivity.ensurePerms`
- `Service.startForeground` bitmap regains `LOCATION` once `ACCESS_FINE_LOCATION`
  is in the perm set
- `MainActivity` needs the runtime perm prompt (group with `RECORD_AUDIO`
  and `POST_NOTIFICATIONS`)

### 3.3 SPA UI surfaces still confusing on Android (residual)

Out of scope for phase 3 polish but worth a sweep:
- **Channels page** form: `output_device_id` selector + TX delay/gain/repeats
  fields are meaningless until phase 5 ships. Should be hidden behind
  `{#if !isAndroid()}`.
- **Channels page** dashboard summary: `Packets TX` counter is decorative
  on Android (always 0). Could be hidden or labeled "TX disabled".
- **Beacons page**: configuring beacons that won't actually transmit until
  phase 5 is confusing. Either hide the page or surface an "enable in phase 5"
  banner.
- **GPS page**: shows config UI for a feature that doesn't exist yet on
  Android. Either hide until phase 4 or surface "phase 4" banner.
- **AGW page** + **iGate page** + **Digipeater page**: most of these depend
  on TX, which doesn't exist. Same treatment.
- **AudioDevices form modal**: `device_path` field is required-validated but
  meaningless. Pre-fill `android-default` and lock to read-only on Android.

The pattern is established (`web/src/lib/platform.js` + `{#if !isAndroid()}`).
A focused 2-3 hour SPA pass would close all of these.

### 3.4 Modem level meter calibration

The new `DeviceLevelUpdate` publication is computed from post-gain samples.
The peak/rms thresholds in `audio::ingest` (`peak >= 32000 → clipping`) match
the desktop modem path. Verify on T865 that the SPA's level meter renders
sensibly; if it pegs at -60dB constantly, the dBFS conversion needs a real
recording to calibrate against.

### 3.5 Restart-time UX

Phase 2's supervisor restart works but `supervisorRestart()` (in
`GraywolfService.kt`) restarts BOTH the modem cdylib AND the Go child on every
Go-child crash. This means:
- Audio capture pauses for ~100-200ms during restart
- All in-flight modem state (DCD, configured channels) resets
- WebView sees a brief connection failure, auto-reloads (already handled by
  `MainActivity.didReloadOnError`)

For phase 5+, consider whether modem restart should be coupled to Go restart
or independent. Independent restart would preserve audio continuity but
requires modem cdylib to handle Go-disconnect cleanly (currently the modem
exits its IPC accept loop on disconnect, but that path isn't fully tested).

### 3.6 Logbuffer not yet wired

`GRAYWOLF_LOG_DB` env var is unconsumed. Desktop logbuffer wires itself via
configstore at runtime; Android-specific override would require a new
`app.Config.LogBufferDBPath` field. Defer to phase 4 or phase 6.

### 3.7 Diagcollect / flare

`pkg/diagcollect/review_android.go` doesn't exist. Phase 7-ish per design
doc. Diagnostics surface degrades to logcat-only on Android, which is
acceptable for the bench-only audience phase 3 reaches.

## 4. Architectural decisions worth carrying forward

### 4.1 Bearer token rotation cadence

**Decision:** per-`GraywolfApp` lifetime (cold start), NOT per-Go-child-restart.

**Why:** rotation per Go restart would require a Service→Activity broadcast +
`WebView.reload()` + SPA bridge cache invalidation. The threat model
(per-launch token preventing other apps from minting valid headers) is
preserved by cold-start rotation. The cached `androidBridge.js` token therefore
stays valid for the lifetime of the WebView session, which matches the Service
lifetime.

Phase 5+ should not "fix" this to match the original spec wording without
coordinating the WebView reload.

### 4.2 modembridge connect-only mode

**Decision:** `Config.ExistingSocket` is a tristate-ish: empty = fork-as-before,
non-empty = dial-only.

**Why:** preserves desktop semantics with zero behavior change while letting
Android skip the fork+readiness handshake. Could be extended for any future
in-process embedding (testing harnesses, CI fixtures).

### 4.3 Bearer middleware bypasses static SPA paths

**Decision:** wrapper applies bearer only to `/api/`, `/ws/`, `/tiles/`.
Everything else (SPA shell + assets) stays public.

**Why:** chicken-and-egg avoidance. WebView's first navigation needs to load
the SPA HTML to install the bridge-aware fetch wrapper. The threat model is
preserved: there's no sensitive data behind `/`, `/assets/*`, `/favicon*`,
or `/index.html` — they're the same bytes shipped to every desktop install.

### 4.4 Foreground Service type matches access perm landscape

**Decision:** Phase 3 declares only `microphone` FGS type. `connectedDevice`
and `location` come back when their access perms (USB_DEVICE,
ACCESS_*_LOCATION) ship in the corresponding phases.

**Why:** Android 14 enforces "FGS type X requires access perm Y to be granted
or declared". Declaring an unused FGS type → SecurityException at
`startForeground`. Don't declare ahead.

### 4.5 PositionLog default-on for Android

**Decision:** seed an enabled `PositionLogConfig` row on first cold boot
when `Platform == "android"`.

**Why:** desktop default-off protects SD-card-based Pi installs from write
amplification. Android internal storage doesn't share that constraint, and a
disabled position log wipes the live map on every process restart — a
confusing operator-visible failure mode that masks an otherwise-working RX
pipeline.

### 4.6 ConfigState mapping is degenerate (single channel)

**Decision:** modem cdylib stores one input_device_id + one channel_id
captured from the most recent `ConfigureChannel` IPC. No per-channel
multi-pipe routing.

**Why:** there's exactly one audio source on Android (`AudioPump` →
`AudioSource.MIC`). Multi-channel routing is meaningless with a single
input pipe. Phase 5+ may grow this into a real per-device pipe table when
output device / TX support lands and multi-radio configurations become
relevant.

## 7. Open questions for the design author

1. **TX/PTT proto path** (phase 5): should USB device permission be auto-attach
   via manifest `<intent-filter android:name="android.hardware.usb.action.USB_DEVICE_ATTACHED"/>`
   plus per-vendor `device-filter.xml`, or interactive runtime prompt every
   time the operator plugs the radio in? Auto-attach is the better UX but
   harder to test (need real Digirig/AIOC IDs).

2. **GPS Service split** (phase 4): is it cleaner to fold the
   `LocationManager` listener into `GraywolfService` (single FGS, single
   notification, simpler lifecycle) or to introduce a separate `GpsService`
   with its own `FOREGROUND_SERVICE_TYPE_LOCATION`? Single Service is simpler
   but couples the lifecycles; separate Services scale better if more
   foreground subsystems land.

3. **OutputDevice configuration model**: phase 5 needs to wire AudioTrack to
   a configurable output device. On Android the operator doesn't pick a
   device — they pick "send to selected USB audio output" or "system
   default". Should the SPA's `output_device_id` field be replaced with a
   binary "use USB if attached" toggle, or expose the system audio routing
   API (`AudioManager.setCommunicationDevice`, API 31+)?

4. **Multi-channel scaling**: phase 3's `config_state` is single-channel.
   Phase 5+ might want multi-radio (Digirig + AIOC concurrently). Does the
   design accommodate that, or is it explicitly single-radio for phase 5
   too? If multi, modem cdylib needs per-channel pipe routing inside the
   single AudioPump → JNI hand-off.

5. **OTA update strategy**: Play Store is the assumed channel
   (`updatescheck` is gated off on Android). But the operator workflow for
   sideload + auto-update from a private update server (e.g. for ham radio
   field deployments without Play Store access) deserves a design pass —
   currently undefined.

6. **Diagnostic/flare path** (phase 7?): Android logcat is per-app and gets
   recycled. Should there be an in-app log buffer (mirror of Phase 6's
   `pkg/logbuffer` desktop equivalent) plus a "submit diagnostic bundle"
   path that emails or HTTPS POSTs the rolling buffer to an operator-
   designated endpoint?
